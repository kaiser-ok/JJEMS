00

執行摘要

JJEMS 的競爭主張
核心主張

傳統 EMS 是專案，JJEMS 是 EMS 量產系統。每個案場仍可高度客製，但客製不再靠重寫，而是靠資料化、模板化、模組化與快速前端封裝。

工商業儲能案場的 EMS 往往需要長時間客製：設備廠牌不同、通訊協定不同、點位定義不同、單線圖不同、電價與需量規則不同、業主想看的 KPI 也不同。 JJEMS 的機會，是把這些變動拆成可配置的資料模型與可重用的前端模組，讓前端工程師可以快速組裝客戶要的 EMS，同時讓後端資料接入與策略執行保持標準化。

因此，JJEMS 的差異化不是和大型 SCADA 廠比功能清單，而是以更快的交付節奏、更貼近台灣 C&I 場景的策略模板，以及 Battery-Aware 智慧調度，建立「案場快速生成」的護城河。

值得強調的是：「交付快」本身就是優勢，而非廉價的代名詞。更快上線代表業主更早取得節費與效益、導入成本更低、ROI 更快；對我們則是更高的交付週轉與毛利。前端結合 AI 的開發速度更是現階段的時間窗——許多傳統廠商內部尚未能快速導入 AI，短期追不上這個落差。速度之上，再疊配置複利、在地深度與 Battery-Aware，才構成長期護城河。

01

市場痛點

EMS 客製化時間長，主要卡在資料與場域差異
Pain 01

每個案場都像重新開發

單線圖、設備階層、控制策略、電價方案、報表欄位、告警分類都會因案場而異，導致 EMS 專案難以複製。

Pain 02

點位與通訊協定是最大摩擦

PCS、BMS、電錶、液冷、消防、除濕、站控與雲端 MQTT 的欄位定義不一致，工程師常需手動整理、測試與修正。

Pain 03

展示很快，真實接入很慢

Dashboard 可以快速做出 Demo，但若缺少資料驗證、斷線處理、控制回覆、稽核與告警規則，仍難以正式交付。

Pain 04

業主買的是效益，不是圖表

業主關心的是節費、需量控制、停機風險、電池壽命、合規與維運效率，EMS 需要把資料轉成可決策的營運價值。

02

產品定位

以 EMS Factory 取代一次性客製專案
把 EMS 從「配角」提升到「戰略高度」

一般 EMS 只是配角——讓業主「看得到數據」。JJEMS 透過模組化與 AI 輔助大幅縮短建置期，把 EMS 變成業主搶市場的戰略工具：更快上線、更快取得效益、在競爭中卡位。而相對於「軟體商去買櫃湊整套」，我們是從硬體端（自有櫃業務 + 上游電芯）向上整合，垂直整合更深、拿得到電芯級資料、毛利不外流。

03

七大競爭優勢

從交付、資料、策略到長期資產價值
Advantage 01

案場快速生成

用模板與設定檔生成 EMS，而不是每案重做。前端結合 AI 的速度被放大成整體交付速度——更快上線等於更快 ROI，不是廉價。

Advantage 02

資料接入產品化

MQTT、Modbus TCP/RTU、電錶、PCS、BMS、液冷、消防點位轉成資料模型，降低客製成本。

Advantage 03

台灣 C&I 場景貼合

內建高壓三段式電價、契約容量、超約罰款、防逆流、防超載、sReg/AFC 等在地策略。

Advantage 04

邊緣與雲端雙模式

同一套 codebase 支援邊緣單站、純 IP 多站與雲地三層，符合不同業主的資安與營運需求。

Advantage 05

Battery-Aware 調度

憑藉與電芯／櫃廠的關係，我們能取得電芯級（cell-level）資料，才做得出真正的 Battery-Aware：把 SoH（電池健康度）、RUL（剩餘可用壽命）、C-rate（充放電倍率）、溫度與故障風險納入長期資產最佳化——這是拿不到電芯資料的一般 EMS 廠商做不到的。

Advantage 06

AI 但保留安全邊界

AI 做查詢、診斷、報表與建議；硬保護由 BMS/PCS/站控規則負責，降低業主疑慮。

Advantage 07

從 Demo 到驗收閉環

不只呈現畫面，也能輸出通訊測試、點位驗證、控制測試、告警測試與 Commissioning Report。

04

資料蒐集模式

讓 EMS 不需要長時間客製開發的配套能力
模式 用途 對競爭力的意義
MQTT V1.4 接 HiTHIUM SCU 或上級 EMS，取得 BMS、PCS、溫濕度、消防、關口表與儲能表資料。 快速完成北向雲端接入，保留原始 payload 供重放與稽核。
Modbus Poller 直接輪詢 PCS、BMS、電錶、液冷、除濕等設備。 支援沒有完整雲端站控的案場，也能做邊緣單站 EMS。
Excel 點位匯入 把廠商 xlsx 點位表轉成 `modbus_points`、`mqtt_field_map` 與告警定義。 把最耗時的人工整理變成半自動流程，縮短開案到上線時間。
Edge Recorder 現場先錄 1 到 2 週資料，檢查方向、比例、缺值、斷線與時間戳。 正式控制前先完成資料可信度驗證，降低驗收風險。
帳單與 CSV 匯入 匯入台電帳單、需量資料、PV 歷史、既有 SCADA 匯出檔。 在設備尚未完整接通前，就能先做節費與 ROI 分析。
外部 API 接氣象、電價、OpenADR/台電派遣、碳排係數等資料。 讓策略不只看設備即時值，也能納入預測與市場訊號。
05

競品對照

避開正面比拼大型 SCADA 功能，主打交付速度與儲能資產優化

Inavitas 類型競品強調 PMS、SCADA、IQ、即時控制、多協定、報表、行動維運與預測最佳化。這些能力成熟，但也意味著導入通常需要較重的工程配置。 JJEMS 應避免只做功能清單追趕，而是主打更快的案場成形速度、更貼近台灣工商業電價與需量場景，以及 Battery-Aware 的儲能專屬價值。

比較面向 一般 EMS / SCADA 廠 JJEMS 主張

資料接入 支援多協定，但常需工程人員逐案設定 把協定、欄位、比例、代碼對照表（例如 0=在線、1=離線）、告警規則全部資料化

產品化路徑

把工程速度變成可複製的商業能力
Phase 1
Configuration First

建立站點、設備、點位、電價、策略、告警與單線圖的配置模型。

Phase 2
Data Ingestion

完成 MQTT V1.4、Modbus poller、原始資料保存與 telemetry 寫入。

Phase 3
EMS Studio

前端用 wizard 組裝客戶 EMS，匯入點位後自動生成初始 dashboard。

Phase 4
Intelligence Layer

導入預測、最佳化、Battery-Aware 調度、AI Copilot 與跨站比較。

07

建議下一步

最能快速形成產品壁壘的五件事
  1. 點位表匯入器：將 MQTT/Modbus xlsx 轉成資料庫設定，先解決客製案場最耗時的整理工作。
  2. MQTT V1.4 ingestion：把 HiTHIUM SCU 上行資料寫入 `mqtt_messages_raw` 與 telemetry tables。
  3. Modbus poller：PCS、BMS、電錶、液冷各完成一版可配置輪詢。
  4. 案場設定 wizard：站點、設備、電價、策略、告警、單線圖一次配置完成。
  5. Commissioning Report：自動產出通訊測試、點位驗證、控制測試、告警測試與交付報告。
結論

JJEMS 的勝負手，是把「快速前端工程」升級成「快速 EMS 交付系統」。當資料接入、點位映射、策略模板與前端模組都能複製，客製案場就不再是交付負擔，而會變成產品的核心優勢。
