# J&J Power EMS · AI Chat 導入效益分析

本文件記錄「將 AI Chat（LLM 對話介面）導入 EMS 儲能能源管理系統」的商業與技術評估，作為產品路線圖的參考依據。

---

## 八大核心效益

### 1. 自然語言操作儀表板
取代點選多層選單，直接用問句拿到答案：
- 「今天到目前節費多少？跟昨天比？」
- 「SYS-B 電芯溫度有沒有異常？」
- 「為什麼 02:47 那筆充電完成？」

降低操作者學習門檻，非工程背景主管也能直接查詢。

### 2. 策略推薦與決策輔助
AI 整合即時資料 + 電價 + 氣象預報後主動給建議：
- 「明天有寒流，廠區負載預計 +15%，建議切換到削峰填谷模式並把 SoC 上限拉到 95%」
- 「下週六台電發布尖峰預警，建議報名 sReg 派遣」
- 預估各策略的明日收益、風險，讓操作員選擇

### 3. 告警智能診斷（RAG 知識庫）
告警觸發時自動帶出：
- 根因分析（這顆 PCS 過去 30 天類似事件 5 次，多發生在高溫午後）
- 處置建議（先檢查 AC-2 空調出風溫度，若 >32°C 則...）
- 對應 SOP / 廠商手冊片段（向量檢索）
- 縮短維運人員從「看到告警」→「處理完」的時間

### 4. 自動報表生成
- 「幫我做一份 4 月的 ESG 月報，含碳減量、ROI、超約次數」
- AI 直接抓資料、產 PDF，不用人工編輯
- 對應 POXA 簡報提到的「碳數位查證」AI 模組概念

### 5. 異常預測（Predictive Maintenance）
- 學習電芯電壓/溫度時序，提前 7–14 天預警 SOH 下降
- 「SYS-A 第 5 模組 SoH 衰退斜率變陡，建議排定檢測」
- 降低意外停機，延長保固理賠成功率

### 6. 多角色介面差異化
同一套後端，不同身份問同樣問題拿不同深度的回答：
- **經營主管**：「這月儲能賺多少？」→ 收益總額 + 同期比 + ROI 進度
- **維運工程師**：同樣問題 → 含 cycle count、各 PCS 效率、有沒有衰退跡象
- **唯讀訪客**：只回公開部分

### 7. 法規/標準合規問答
- 「這次設備變更要符合哪些 IEC 標準？」
- 「台電 sReg 最新規範什麼時候生效？」
- AI 即時對照法規庫（IEC 62443、IEC 61850、ISO 50001、ISO 27001、台電 UL9540A），省掉人工翻文件

### 8. 跨站點對標（Multi-site Benchmarking）
未來多站點：「比較高雄路竹廠 vs 桃園廠近 30 天 PR、節費效率，哪邊優化空間最大？」AI 自動跑統計並給結論，主管快速掌握資產組合表現。

## 商業價值估算

| 項目 | 量化效益 |
|---|---|
| 維運人力 | 減少 30–50%（告警處理時間 30 min → 5 min） |
| 電池壽命 | +20–30%（POXA 實績 + AI 建議避免過充放） |
| 案場收益 | +16%（策略推薦 + 動態調度） |

| 銷售差異化 | 強力賣點 — POXA / 加雲 / Siemens 目前均為 BI 圖表型，無對話介面 |

## 技術架構

```
┌─────────────────────────────────────────┐
│   Web / Mobile EMS (JJEMS SPA)          │
│   ┌─────────────────┐                    │
│   │ 浮動 Chat Widget │                    │
│   └────────┬────────┘                    │
└────────────┼─────────────────────────────┘
             │ WebSocket / HTTP
             ▼
┌─────────────────────────────────────────┐
│   Chat Orchestrator (Node.js / Python)   │
│  ┌──────────┐  ┌─────────────────────┐  │
│  │ Intent   │  │ Tool calling        │  │
│  │ router   │  │ (EMS API / Modbus)  │  │
│  └────┬─────┘  └──────┬──────────────┘  │
│       │               │                  │
│       ▼               ▼                  │
│  ┌──────────────────────────────┐       │
│  │ Claude API / OpenAI API      │       │
│  └──────────┬───────────────────┘       │
│             │                            │
│       ┌─────▼─────┐                     │
│       │ RAG 檢索  │                      │
│       └─────┬─────┘                     │
└─────────────┼───────────────────────────┘
              │
      ┌───────┴────────┐
      ▼                ▼
┌──────────┐     ┌──────────────┐
│ 向量 DB   │     │ 時序 DB       │
│ (pgvector│     │ (TimescaleDB)│
│ /Pinecone)│    │               │
└──────────┘     └──────────────┘
      ▲                ▲
      │                │
 廠商手冊 / SOP    EMS 即時 / 歷史資料
 法規 / 電價規範
```

**關鍵元件**
- **LLM**：Claude 4.X / GPT-4 class（中文理解 + 工具呼叫能力）
- **RAG 知識庫**：廠商手冊、法規、SOP、維修歷史
- **Tool Calling**：AI 能實際呼叫 EMS API（查詢/控制），不只回答
- **時序資料庫**：TimescaleDB / InfluxDB，存電力、電池、電錶資料
- **權限層**：依使用者角色限制可查詢/可執行範圍（RBAC）

多站部署時邊際成本下降（共用向量庫與模型）。

## 建議落地路徑

### Phase 1 — MVP（2–4 週）
- 前端浮動 Chat 視窗
- 5–8 個核心 intent（節費查詢、SoC、告警、策略切換）
- 靜態規則 + 簡單 LLM 回答，不接 RAG

### Phase 2 — 知識強化（4–8 週）
- 建立 RAG 知識庫（手冊、法規、SOP）
- Tool calling 接入真實 EMS API
- 增加策略推薦、告警診斷

### Phase 3 — 預測分析（8–12 週）
- 電池 SOH 預測模型
- 負載 / PV 預測
- 跨站點對標、自動報表

### Phase 4 — 產品化
- 品牌化為「能源 GPT」或類似（對標加雲 EnergyGPT）
- SaaS 計費模式
- 白牌授權給其他 EMS 廠商
